皆さんはClaude Codeのマルチセッション管理をどうしていますか？

Claude Codeの進化により、1つのセッションでこなせるタスクの精度は大幅に上がりました。しかし実際の開発業務では1つのセッションだけでは足りないことの方が多いです。プルリクエストの作成までできたとしても、CIの失敗、コンフリクトの発生、レビューコメントの対応など、後続の細かい作業が続きます。また複数タスクを並列で進行したり、副業など複数プロジェクトに関わるならさらに並列作業が増えていきます。

この問題には世の中には数多くのアプローチが出ており、自分もしばらく試行錯誤を続けていました。最終的にジョブキューベースのオーケストレーションツール[tq](https://github.com/MH4GF/tq)を作ったので紹介します。

https://github.com/MH4GF/tq

## tqとは

tqはジョブキューベースのClaude Codeオーケストレーションシステムです。以下の2つを組み合わせています。

- CLI … 人間向けのTUIと、エージェント向けのコマンド群、バックグラウンドでキューイングとディスパッチを行う
- Claude Code Plugin … エージェントがCLIを活用するためのスキルやコマンド

Claude Codeセッションを呼び出す単位を「アクション」と呼び、アクションをキューに積みます。バックグラウンドで次のアクションをピックアップしてtmux上にClaude Codeセッションを起動します。

![tqのデモ](https://mh4gf.dev/images/tq.gif)

tqの設計思想を4つ紹介します。

### タスク管理の主体はマネージャーエージェント

一つのClaude Codeセッションに、他のセッションを管理するマネージャーの役割を担ってもらいます。人間はTUIで各タスクの進捗を眺めながら、マネージャーエージェントに自然言語で指示を出すだけです。マネージャーエージェントがタスクの分解、進捗状況の確認、トラブル対応を行います。

### ジョブキューによるコンテキストスイッチの削減

tqは、Claude Codeセッションをアクションとして起動する際に対話セッションにするか非対話セッションにするかを選べます。そして対話セッションの同時起動数を制限できます。デフォルトは3です。これにより並列作業数が増えてコンテキストスイッチが大変になることを防ぎ、適度に人間の集中を可能にします。

もちろん、非対話セッションはバックグラウンドで実行可能なため、キューに入ったら即時で起動します。

定型業務は可能な限り非対話セッションにしスケールさせていきたいですよね。tqはまず対話セッションでスキルをブラッシュアップさせながら、安定してきたら非対話セッションに成長させていくことができます。もちろん挙動に違和感があれば対話セッションに戻してデバッグもできます。

### Claude Codeの進化に委ねる

tqはジョブキューの仕組みを提供するだけで、特に専用のワークフロー定義やプロンプトシステムを提供しません。tmuxで `claude` コマンドを引数とともに実行するだけです。そのためClaude Codeのスキルやサブエージェント、Git Worktreeサポート、リモート実行などの仕組みをそのまま使えます。

ハーネスエンジニアリングや業務のスキル化、今後のエージェントの進化についてはClaude Codeに任せ、tqは薄いオーケストレーション層として振る舞います。

### CLIコマンドはエージェントのためのもの

tqはアクションの作成や各種設定、全文検索などのCLIコマンドを提供していますが、それらは全てAIが利用する前提で最適化されています。

- 出力はJSONのみで `--jq` フラグによる絞り込みができる
- `--help` の出力にはエージェント向けのベストプラクティスを含めている
- `tq action create` コマンドは、claudeに渡す指示だけでなくタイトルも必須。タイトルはTUIで表示される。人間が書くとなると面倒だが、AIが書くなら負荷は低い

人間はtqの使い方を知る必要がない、という意図的な設計です。

## tqの使い方

tqは、CLIのダウンロードとClaude Code Pluginのインストールが必要です。

```bash
# CLIインストール
brew install --cask MH4GF/tap/tq

# Claude Code Pluginインストール
claude plugin marketplace add MH4GF/tq
claude plugin install tq@tq-marketplace

# TUIを起動（キューワーカーが自動で動き出す）
tq ui

# claudeを開いて、マネージャーエージェントを起動
claude
/tq:manager
```

tq uiとマネージャーエージェントのペインを横並びにして起動しておくのが普段のスタイルです。  
マネージャーエージェントはtqの使い方を知っているので、あとは自然言語で相談していくだけでOKです。その一つとして、最もよく使うアクション作成コマンドを実行してみましょう。

```
/tq:create-action 機能Xを実装してPRを作成して
```

これでアクションが作成され、Claude Codeセッションが起動します。TUI上でアクションの進捗状況を追うことができます。内部的には、スキルに従って`tq action create` コマンドを実行します。

マネージャーエージェントを介することで、適当な指示からプロンプトチューニングされた指示にブラッシュアップしてくれることも地味なメリットです。

他にも以下のような機能があります。

- スケジュール起動 ... 定期的にアクションをキューに追加します。
- タスクスコープのメモリ ... 過去のアクションの結果を確認できるため、コンテキストを引き継げます。
- プロジェクト単位のフォーカス ... 稼働時間外は特定プロジェクトのアクションを実行しない、みたいな制御ができます。 

## 使い方の例

tqはtqでタスク管理して開発していますし、私は普段の業務もtqで行っています。どのように日常的に利用しているか、具体的な場面を挙げてみます。


### 作業中の割り込みをキューに逃がす

Claude Codeで対話的に作業していて「あ、これも直さないと」と気づいたとき、以前なら`/fork` でセッションを切り替えるか、メモして後回しにしていました。

tqなら `/tq:create-action これを直して` で投げるだけです。今の作業は中断せず、別のClaude Codeセッションが非同期で対応してくれます。タスクの過去ログを引き継ぐので、文脈を説明し直す必要がありません。


### GitHub通知の自動処理

5分ごとにGitHub通知をチェックするスケジュールを設定しています。通知内容に応じてアクションを切り替えます。コンフリクトが起きたら修正、CIが落ちたら修復、レビューコメントが来たら対応を検討して承認待ちまで進めておく。こういった煩雑な作業がバックグラウンドで自動的に進みます。

このワークフローはサンプルとして公開しています。

https://github.com/MH4GF/tq/tree/main/.claude-plugins/gh-ops/README.md

`/tq:create-action https://github.com/MH4GF/tq/tree/main/.claude-plugins/gh-ops/README.md をセットアップして` と指示してみてください。  

### メール対応の自動化

[gogcli](https://github.com/steipete/gogcli)を利用し、Gmailのインボックスを定期巡回するスケジュールも動かしています。返信ドラフトの作成、unsubscribeの実行、それぞれ別のactionとしてキューに積まれます。煩雑なメール対応がほぼ自動化されました。

### 定型作業のループ(Ralph Loop)

定型作業のスキル`/repetitive-task` を作り、その末尾に `/tq:create-action /repetitive-task` を仕込んでおくと、一つのactionが完了するたびに次のactionが生成され、コンテキストをクリアしながらループし続けます。いわゆるRalph Loopです。

公式のralph-loopプラグインとの違いは、作業ごとに新しいセッションとして作成されるためコンテキストウィンドウ圧迫による性能低下がないことです。

### 一日の振り返り

`/daily-report` スキルを作り、tqを使ってその日のタスク進捗を元にObsidianに日報を作成してもらいます。基本的にtqを経由してClaude Codeを操作することでタスクの情報がメモリに記録されていくので、その日の作業ログを簡単にまとめられるようになります。
これも1時間に一回スケジュール実行するようにしているので、いつでも日報が準備できています。

## 既存ツールや機能との違い

Claude Codeの並列実行を支援するツールはこの領域にいくつか出てきています。tqの立ち位置を整理しておきます。

### セッションマネージャー系

[claude-squad](https://github.com/smtg-ai/claude-squad)や[ccmanager](https://github.com/kbwo/ccmanager)は、tmux + git worktreeで複数セッションを並列管理するツールです。TUI上でセッションの起動・切り替え・監視ができて、「今この瞬間に複数セッションを走らせたい」というニーズに強いです。

tqはこの領域に近く、tmuxベースで複数セッション管理することは同じです。違いはジョブキューで並列実行数を制限できること、色んな方法で気軽にアクションを積むことができることです。

### ワークフローオーケストレーター系

[TAKT](https://github.com/nrslib/takt)は、YAMLでワークフロー（Piece/Movement）を定義し、AIエージェントを決められた手順に沿って「強制」するアプローチです。再現性が高く、CI/CDパイプラインとの統合に向いています。

tqは専用のワークフロー定義言語を持たず、Claude Codeに自然言語で指示を出すという普段の使い方をそのまま延長する設計です。

### Claude Codeのビルトイン機能

[Agent Teams](https://docs.anthropic.com/en/docs/claude-code/agent-teams)は、上位AIがタスクの分配・統合を行う構成です。
tmuxで複数セッションを管理する点はtqと同じですが、Agent Teamsは一つの明確なゴールに向かって進むため、1つのセッションで複雑な作業を完遂させることに向いています。複数の関連しないタスクを扱うのは不慣れです。
tqは常に起動し複数タスクを並列で進めることに向いているため、エンジニアの日々の作業フローに近い感覚で管理できます。もちろんtqで起動するアクションの内部でAgent Teamsを利用することもあるでしょう。

`/loop`は、現在のセッション内でプロンプトを定期実行するコマンドです。`/loop 30m デプロイが完了したか確認して` のように使います。手軽ですが、セッションを閉じるとタスクが消えます。短期的なポーリング向けです。

`/schedule`は、Anthropicのクラウド上で定期実行するコマンドです。マシンがオフでも動き、`/schedule daily 毎朝PRをレビューして` のように使えます。ただし毎回フレッシュな環境での実行になるため、ローカルファイルへのアクセスや過去のタスク履歴の引き継ぎはできません。最小インターバルも1時間です。

tqはこの2つの間を埋めるポジションにあります。ローカル環境で動き、SQLiteにタスク履歴が永続化され、過去のアクション結果を引き継げます。セッションを閉じてもキューは残り、`tq ui`を起動すればworkerが再開します。

## おわりに

tqのアプローチに至る前、最初はcrontab + tmuxでマネージャー役のClaude Codeにサブセッションを`tmux send-keys`で管理させるところから始めました。しかし双方向の同期通信ではマネージャーが忙しいときに報告が漏れるという問題にぶつかり、間にキューを挟むアプローチに移行しました。振り返ると、解決すべき本質的な問題は「複数セッションをどう並列に動かすか」ではなく、「人間のコンテキストスイッチをどう減らすか」だったと思います。

tqを使うようになってから、GitHub通知の処理やCI修正、メール対応といった雑務の大半が自動化され、重要なタスクに向き合う時間を大幅に増やせるようになりました。
